## Hallazgos detallados

### A01 — Broken Access Control · 🔴 CRITICAL

**H-01.1 🟢 Token de sesión admin predecible y sin validación server-side — CERRADO 2026-04-19**

Archivo: [lib/admin-auth.ts:16](lib/admin-auth.ts) (antes del fix):
```ts
const token = Buffer.from(`${Date.now()}-${Math.random().toString(36)}`).toString('base64')
```

Problemas:
- `Date.now()` + `Math.random()` generan tokens adivinables (no crypto-safe)
- Base64 no es cifrado — reversible en 1 segundo
- Peor aún: `isAdminAuthenticated()` solo validaba `!!session?.value` — cualquier cookie no vacía pasaba

**Exploit pre-fix:** cualquier atacante con `Cookie: preenvios_admin_session=anything` obtenía acceso total a `/api/admin/*`.

**Fix aplicado (2026-04-19):**

Implementación HMAC-signed stateless token (sin dependencias nuevas, solo `node:crypto`):

1. ✅ **Formato del token:** `<random32-base64url>.<expires-at-unix-ms>.<hmac-sha256-base64url>`
   - `random`: 32 bytes de `crypto.randomBytes` → entropía crypto-safe
   - `expires-at`: timestamp absoluto de expiración (validado server-side)
   - `hmac`: HMAC-SHA256 de `<random>.<expires-at>` con `ADMIN_SESSION_SECRET`

2. ✅ **`createAdminSession()`:** genera los 3 campos, firma con HMAC y persiste el token completo en cookie HttpOnly.

3. ✅ **`isAdminAuthenticated()`:** parsea la cookie en 3 partes, recomputa HMAC y compara con `crypto.timingSafeEqual` (previene timing attacks), valida expiración. Si cualquier check falla → false.

4. ✅ **Duración de sesión reducida de 24h → 4h** (cumple checklist A07 "timeout admin ≤ 4 horas").

5. ✅ **Nueva env var `ADMIN_SESSION_SECRET`** (32 bytes hex). Rotarla invalida todas las sesiones vivas → mecanismo de revocación simple sin tabla en DB.

6. ✅ **Sin fallback:** si `ADMIN_SESSION_SECRET` falta, `sign()` lanza y `verifySignature()` captura y retorna false → login imposible en vez de downgrade silencioso a tokens débiles.

7. ✅ **Acción manual en Vercel completada 2026-04-19:** `ADMIN_SESSION_SECRET` agregada en los 3 entornos (Production/Preview/Development). Panel admin probado en producción tras el redeploy — login y persistencia de sesión funcionan correctamente.

**Por qué HMAC-signed stateless vs tabla en DB:**
- No requiere migración ni round-trip a Supabase en cada request → dashboard admin más rápido
- Revocación vía rotación del secret (suficiente para 1 admin user)
- Se reemplaza por Supabase Auth en Fase 4.4.B de todos modos — no vale la pena construir infra de sesiones temporal en DB

**H-01.2 🟢 `NEXT_PUBLIC_CRON_SECRET` expuesto en bundle cliente — CERRADO 2026-04-19**

Archivo: [app/[locale]/admin/panel.tsx:61](app/%5Blocale%5D/admin/panel.tsx#L61) (antes del fix):
```tsx
const res = await fetch('/api/admin/dashboard', {
  headers: { authorization: `Bearer ${process.env.NEXT_PUBLIC_CRON_SECRET || ''}` }
})
```

Problema:
- Variables con prefijo `NEXT_PUBLIC_*` se inlinean en el JavaScript del bundle que se envía al navegador
- Cualquier visitante de `/es/admin` puede abrir DevTools → Sources y leer el valor
- Con ese secreto un atacante puede invocar `/api/scrape` y disparar scrapers, forzar 3-strikes failure y envenenar datos de todos los operadores

**Fix aplicado (2026-04-19):**
1. ✅ `app/api/admin/dashboard/route.ts` ahora valida con `isAdminAuthenticated()` (session cookie) en vez de `CRON_SECRET`. Eliminado el import de `NextRequest` y el param `request`.
2. ✅ `app/[locale]/admin/panel.tsx:61` — removido el header `authorization`. El fetch usa la cookie HttpOnly que el navegador adjunta automáticamente.
3. ✅ `.env.example` actualizado: comentario explícito indicando que `CRON_SECRET` NUNCA debe llevar prefijo `NEXT_PUBLIC_` y es exclusivo de `/api/scrape`.
4. ✅ **Acción manual en Vercel completada 2026-04-19:** `NEXT_PUBLIC_CRON_SECRET` eliminada de los 3 entornos y `CRON_SECRET` rotada (nuevo valor, solo server). Login admin verificado en producción tras el cambio.

### A02 — Cryptographic Failures · 🔴 CRITICAL

**H-02.1 🟢 `ADMIN_PASSWORD` comparado en texto plano — CERRADO 2026-04-19**

Archivo: [lib/admin-auth.ts:11-13](lib/admin-auth.ts) (antes del fix):
```ts
export async function validateAdminLogin(email: string, password: string): Promise<boolean> {
  return email === process.env.ADMIN_EMAIL && password === process.env.ADMIN_PASSWORD
}
```

Problema:
- Si el `.env.local` filtra (backup comprometido, log leak, git error, misconfiguración Vercel), el password queda expuesto
- No hay bcrypt/argon2 hashing
- Supabase Auth en Fase 4.4.B mitiga esto, pero hasta entonces es vulnerable

**Fix aplicado (2026-04-19):**
1. ✅ Instalado `bcryptjs@^3` + `@types/bcryptjs` (dev). `npm audit`: 0 vulnerabilidades.
2. ✅ `validateAdminLogin` usa `bcrypt.compareSync(password, process.env.ADMIN_PASSWORD_HASH)` con short-circuit si faltan env vars (return false en vez de lanzar).
3. ✅ `.env.example` actualizado — variable renombrada de `ADMIN_PASSWORD` → `ADMIN_PASSWORD_HASH` con comentario explicando cómo generar el hash.
4. ✅ **Acción manual en Vercel completada 2026-04-19:**
   - `ADMIN_PASSWORD` (plaintext) eliminada de los 3 entornos.
   - `ADMIN_PASSWORD_HASH` (bcrypt factor 12) agregada en los 3 entornos.
   - Login admin verificado en producción con la password original tras el redeploy.

5. ℹ️ **Sin fallback a plaintext:** si `ADMIN_PASSWORD_HASH` falta, el login devuelve 401 siempre — intencional para evitar downgrade silencioso.

**Nota Supabase Auth (Fase 4.4.B):** al migrar, esta función se deprecia entera. Mientras tanto, bcrypt factor 12 con hash en env var es el estándar aceptable.

**H-03.4 🟡 Webhook Twilio NO valida firma HMAC**

Archivo: [app/api/whatsapp/webhook/route.ts:60-81](app/api/whatsapp/webhook/route.ts):
```ts
const body = await request.formData()
const from = body.get('From')
const text = body.get('Body')
// ... procesa sin verificar X-Twilio-Signature
```

Problema: cualquier atacante puede POST al webhook fingiendo ser Twilio, disparar lógica del bot, abusar del endpoint para enumerar corredores, o inyectar respuestas falsas en la cola.

Mitigación parcial: el bot no ejecuta acciones con efectos secundarios graves (solo consulta tasa + responde). Pero el riesgo sube si en el futuro se agrega logging de mensajes o integración con Supabase.

**Fix requerido** (TROUBLESHOOTING/16 ya documenta el patrón):
```ts
import { validateRequest } from 'twilio'
const signature = request.headers.get('x-twilio-signature') || ''
const url = process.env.TWILIO_WEBHOOK_URL!
const params = Object.fromEntries(body.entries())
if (!validateRequest(process.env.TWILIO_AUTH_TOKEN!, signature, url, params)) {
  return NextResponse.json({ error: 'invalid signature' }, { status: 401 })
}
```

### A04 — Insecure Design · 🟡 MODERATE

**H-04.1 🟡 Sin rate limiting en endpoints públicos**

Endpoints afectados:
- `POST /api/suscripcion-free` → permite spam masivo de altas + floods vía Resend
- `POST /api/contactos` → permite flood de mensajes en la tabla `contactos`
- `POST /api/admin/auth` → brute force ilimitado (también A07)

**Fix requerido:**
- Instalar `@upstash/ratelimit` + `@upstash/redis` (plan free cubre ~10K req/día)
- Límites sugeridos:
  - `/api/suscripcion-free`: 3 req / 10 min por IP
  - `/api/contactos`: 5 req / 10 min por IP
  - `/api/admin/auth`: 5 intentos / 15 min por IP + respuesta `429 Too Many Requests`

**H-04.2 🟢 Double opt-in implementado**

`/api/suscripcion-free` crea suscripción con `confirmado: false` y solo activa tras GET con token válido ✓

### A05 — Security Misconfiguration · 🟡 MODERATE

**H-05.1 🟡 Faltan headers de seguridad globales**

[next.config.ts](next.config.ts) no configura headers. El sitio hoy sirve sin:
- `Strict-Transport-Security` (HSTS)
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Content-Security-Policy` (CSP)
- `Referrer-Policy: strict-origin-when-cross-origin`

**Fix requerido** (TROUBLESHOOTING/18 ya lo describe). Agregar a `next.config.ts`:
```ts
async headers() {
  return [{
    source: '/:path*',
    headers: [
      { key: 'Strict-Transport-Security', value: 'max-age=63072000; includeSubDomains; preload' },
      { key: 'X-Content-Type-Options', value: 'nosniff' },
      { key: 'X-Frame-Options', value: 'DENY' },
      { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
      { key: 'Content-Security-Policy', value: [
        "default-src 'self'",
        "script-src 'self' 'unsafe-inline' https://www.googletagmanager.com",
        "style-src 'self' 'unsafe-inline' https://fonts.googleapis.com",
        "font-src 'self' https://fonts.gstatic.com",
        "img-src 'self' data: https://flagcdn.com https://cdn.brandfetch.io https://www.google-analytics.com",
        "connect-src 'self' https://*.supabase.co https://www.google-analytics.com",
        "frame-ancestors 'none'",
      ].join('; ') },
    ],
  }]
}
```

**H-05.2 🟡 Mensajes de error podrían filtrar info de schema**

Varios `catch` devuelven `error.message` directo de Supabase, que puede incluir nombres de columnas o constraints. No es alto riesgo pero recomendable sanitizar a `"Error interno"` en producción.

### A07 — Identification and Authentication Failures · 🔴 CRITICAL

**H-07.1 🟢 Admin login sin rate limit ni lockout — CERRADO 2026-04-19**

Pre-fix: `/api/admin/auth` aceptaba intentos ilimitados. Un atacante con diccionario podía probar miles de passwords sin consecuencias, anulando parcialmente el fix de bcrypt (H-02.1).

**Fix aplicado (2026-04-19):**

Rate limit Supabase-backed (sin dependencia externa tipo Upstash/Redis):

1. ✅ **Migración 006** [`supabase/migrations/006_admin_login_attempts.sql`](../supabase/migrations/006_admin_login_attempts.sql): tabla `admin_login_attempts` con `ip_hash`, `email`, `success`, `attempted_at`. RLS activo + políticas deny-all para anon. Índice compuesto `(ip_hash, attempted_at DESC)` para lookups rápidos.

2. ✅ **`lib/rate-limit.ts`** — nuevo módulo:
   - `getClientIp()`: extrae IP desde `x-forwarded-for` (Vercel) con fallback a `x-real-ip`
   - `hashIp()`: SHA-256 de la IP (privacy — no guardar plaintext)
   - `checkAdminLoginRateLimit()`: cuenta intentos fallidos del `ip_hash` en los últimos 15min. Si ≥5 → deniega con `retryAfterSeconds = 900`
   - `recordAdminLoginAttempt()`: inserta cada intento (success + fail) — también sirve de audit trail (bonus parcial a H-09.2)

3. ✅ **`app/api/admin/auth/route.ts`**: chequea rate limit antes de validar credenciales. Si bloqueado → HTTP 429 con header `Retry-After: 900`. Registra el intento siempre (pasó validación o no, pasó rate limit o no).

4. ✅ **Configuración:** 5 intentos fallidos por IP en 15 min. Ajustable vía constantes al tope de [`lib/rate-limit.ts`](../lib/rate-limit.ts) si el comportamiento resulta muy estricto/laxo tras uso real.

5. ✅ **Fail-open ante errores de Supabase:** si `count()` o `insert()` fallan (DB caída, red), el login procede. Decisión consciente — el admin no debe quedar bloqueado por fallos de infra. Los errores se loggean para monitoreo (Sentry pendiente — H-09.1).

6. ⚠️ **Acción manual en Supabase antes del próximo deploy:**

   Ejecutar en Supabase SQL Editor el contenido de `supabase/migrations/006_admin_login_attempts.sql` (copy-paste ready). Verificar con:
   ```sql
   select count(*) from admin_login_attempts;  -- debe retornar 0
   ```

**Por qué Supabase-backed en vez de Upstash/Redis (el audit inicial recomendaba Upstash):**
- No requiere signup/config de nuevo servicio externo
- No requiere env vars adicionales (reusa `SUPABASE_SERVICE_ROLE_KEY`)
- Latencia aceptable para un endpoint de login (no es hot-path de lectura)
- Bonus: la tabla sirve como audit trail del login (H-09.2 parcial)
- Si volumen crece, migrar a Upstash es un cambio localizado en `lib/rate-limit.ts`

**H-07.2 🟡 Cookie `secure` condicionada a NODE_ENV**

Archivo [lib/admin-auth.ts](lib/admin-auth.ts):
```ts
secure: process.env.NODE_ENV === 'production',
```

Correcto para desarrollo local (HTTP) pero fiar del env var es frágil. Un deploy de preview sin `NODE_ENV=production` enviaría cookie sin `Secure`. Hardcodear `secure: true` tras deploy inicial.

**Nota post-fix H-01.1:** la sesión ahora dura solo 4 horas (antes 24h), reduce la ventana de exposición si una cookie se intercepta sobre HTTP por error de config.
